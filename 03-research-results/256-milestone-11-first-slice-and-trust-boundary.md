# Research Result 256 - Milestone 11 First Slice and Trust Boundary

## Recommended first slice

```text
execution attempt
-> verification run
-> return-bundle manifest
-> artifact provenance
```

Deferred within Milestone 11:

```text
governed-operation read model
connector invocation telemetry
```

## Why this is the minimum coherent slice

The four selected families form one complete evidence chain:

1. an approved implementation or resumption attempt starts;
2. exact verification commands run against bound code and input state;
3. generated or frozen artifacts receive provenance assertions;
4. one immutable return manifest freezes the complete evidence inventory;
5. human audit and owner acceptance continue in Markdown.

Removing any family leaves a proven Northstar pain unresolved:

- without execution attempts, retries and terminal state remain ambiguous;
- without verification runs, timeout, blocking, failure, and success remain mixed in prose;
- without return manifests, completeness still requires directory archaeology;
- without provenance, hashes cannot be efficiently traced to inputs, code, or producing runs.

Adding the governance read model or connector telemetry would widen the first slice into unrelated operational concerns.

## Authority boundary

```text
Operational database:
structured state for the four accepted families

Markdown:
human decisions, specifications, audits, verdicts, owner acceptance, and project intelligence

Immutable files:
return evidence, raw verification output, generated artifacts, and frozen reports

Git:
repository commits and tracked-state facts

Governance JSONL and immutable operation evidence:
governed mutation authority and history
```

The operational database may reference these authorities but cannot replace or rewrite them.

## Trust zones

### Zone 1 - Human and canonical authority

Owners and governed reviewers accept decisions, packets, audits, and completion records. These actions remain outside the operational database's unilateral authority.

### Zone 2 - Typed operational service

The only component permitted to mutate the first-slice records. It enforces:

- project scope;
- legal state transitions;
- idempotency keys;
- stable external IDs;
- transaction boundaries;
- privacy minimization;
- evidence-pointer validation;
- audit-event creation where required.

### Zone 3 - Database

Stores accepted structured operational state. It receives no direct agent or MCP credentials.

### Zone 4 - Artifact and evidence files

Stores immutable bytes. The service records references and hashes but does not ingest the bytes into relational fields.

### Zone 5 - Agents and MCP clients

May call approved typed tools later. They cannot issue arbitrary SQL, select unrestricted records, or bypass project and authority checks.

## Minimum typed capabilities for later design

```text
reserve_execution_attempt
start_execution_attempt
finish_execution_attempt
record_verification_run
freeze_return_manifest
record_artifact_provenance
verify_artifact_hash
get_execution_evidence_graph
list_project_attempts
```

Names are conceptual. Packet 016B may revise them before implementation.

## Mutation rules

### Execution attempt

- reservation and start may be separate only if recovery requires it;
- one attempt binds one project, packet/task, starting commit, and allowed scope;
- terminal state is immutable;
- retries create new attempts linked to predecessors;
- success does not mean audit acceptance.

### Verification run

- one record per exact command execution;
- timeout, blocked, transport failure, test failure, and pass are distinct;
- reruns create new IDs;
- raw stdout and stderr remain files where retained;
- summarized counts and bounded error classifications may be structured.

### Return-bundle manifest

- freezing binds an exact evidence inventory and per-file hashes atomically;
- frozen manifests are immutable;
- corrected or expanded evidence creates a new manifest version;
- rejected manifests remain queryable.

### Artifact provenance

- logical artifact identity and immutable byte-object identity remain distinct;
- identical bytes generated by different runs may share a byte hash but retain separate provenance assertions;
- artifact bytes remain files;
- corrections supersede rather than overwrite assertions.

## Cross-system references

The first slice may reference:

```text
project ID
packet or governed task ID
repository identity
repository commit hash
scoped storage root ID
relative path
canonical Markdown record ID
canonical audit or owner-acceptance ID
```

References do not grant mutation authority over those systems.

## Project isolation

Every first-slice record is project-scoped. Typed reads require an explicit project ID. Cross-project joins are unavailable to ordinary agent-facing tools unless a later portfolio-level capability is explicitly approved.

## Privacy boundary

Store only:

- governed identifiers;
- lifecycle state;
- bounded timestamps;
- hashes;
- normalized command or verification kind;
- summary counts;
- scoped storage references;
- bounded error codes;
- lineage references.

Do not store:

- prompt bodies;
- arbitrary command arguments containing secrets;
- environment dumps;
- artifact contents;
- unrestricted stdout/stderr;
- customer payloads;
- credentials or tokens.

## Failure boundaries

- database failure cannot change Git, Markdown, governance JSONL, or evidence-file authority;
- file-write failure after a database reservation leaves an explicit incomplete state;
- database commit after file freeze must verify the exact frozen inventory and hashes;
- unknown state remains recovery-required;
- service retry uses idempotency keys and never creates silent duplicate terminal records.

## Deferred families

### Governed-operation read model

Defer until the first slice proves the database, migrations, backup, restore, typed service, and operational maintenance burden. File/event queries remain authoritative and sufficient for now.

### Connector invocation telemetry

Defer until connector contract improvements and a privacy-reviewed telemetry policy are available. Severe incidents continue to be preserved in bounded audits and existing logs.

## Conclusion

The four-family slice is small enough to prove completely and useful enough to reduce repeated Northstar pain. It does not require moving canonical truth or artifact bytes into Postgres.
