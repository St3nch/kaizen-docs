## Context

Milestones 1 through 4 proved the first complete governed Kaizen loop:

```text
canonical project context
-> source summary
-> claim
-> decision
-> spec
-> audit
-> task packet
-> implementation
-> tests and completion evidence
-> governed task-packet amendment
-> governed current-state amendment
-> closure audit
```

The platform implementation now supports both first-time promotion and bounded same-note amendment. The accepted decisions remain sound, but several planning specifications and maturity labels lag behind the implemented and audited behavior.

Result 064 found six blockers before the consolidated v0.2 project standard can become authoritative:

1. incomplete normative amendment event fields;
2. promotion-centric workflow prose;
3. inconsistent spec maturity labels;
4. ambiguous recovery terminology;
5. unresolved durable human actor identity;
6. unresolved backup and remote posture.

This decision proposes one proportional reconciliation rather than treating each wording defect as a separate architecture exercise.

## Decision

If accepted, Kaizen will adopt the following first-slice contract reconciliation for v0.2.

### 1. Implemented action vocabulary

The implemented and accepted v0.2 mutation actions are:

```text
promote
amend
```

The following values remain reserved planning vocabulary and are not executable capabilities:

```text
supersede
correct
rollback
```

A reserved action must not be described as available until a separate accepted decision, specification, implementation, hammer-test pass, and governed return audit exist.

### 2. Immutable operation evidence is normative

For every governed mutation, the immutable operation directory is authoritative review evidence before execution.

A first-time promotion plan binds at least:

```text
operation ID
packet ID where applicable
source SHA-256
canonical candidate SHA-256
validation result SHA-256
destination
note identity and type
review and authority transition
required canonical dependencies
platform and vault Git binding
governance-log binding
plan SHA-256
```

An amendment plan additionally binds:

```text
prior canonical SHA-256
original staged candidate SHA-256
normalized canonical candidate SHA-256
reviewed diff SHA-256
change summary
continuity fields
```

Execution approval binds the exact immutable plan SHA-256. Approval evidence binds the approved actor, basis, plan, candidate, prior bytes where applicable, reviewed diff where applicable, and validation run.

### 3. Amendment is a bounded mutation, not generalized overwrite

The accepted amendment slice changes exactly one existing canonical Markdown note at its existing destination.

It preserves:

```text
id
type
project
created
```

It also preserves review status and authority unless a separately accepted transition is explicitly supported by the governing spec and reviewed plan.

Amendment does not authorize:

```text
delete
move
rename
general overwrite
multi-note transaction
supersedence
correction
rollback execution
```

### 4. Validation has two layers

#### Intrinsic note validation

Checks the note itself:

- UTF-8 and flat YAML;
- required fields and enums;
- ID and type consistency;
- required body sections;
- lifecycle and authority rules;
- prohibited private or structural content.

#### Operation-context validation

Checks the mutation in context:

- fixed roots and path confinement;
- destination placement;
- canonical dependency and relationship resolution;
- local Markdown-link resolution;
- prior/source/candidate/diff/validation hashes;
- packet and operation binding;
- Git branch, HEAD, and clean state;
- governance-log state;
- approval freshness;
- filesystem safety.

Intrinsic canonical validation alone does not authorize mutation.

### 5. Event phases and operation outcomes

The canonical event phases for new v0.2 events are:

```text
intent
committed
failed
recovered
```

Their meanings are:

- `intent` - durable pre-mutation record;
- `committed` - normal successful terminal record;
- `failed` - terminal record showing the approved operation did not produce an accepted canonical result;
- `recovered` - terminal record showing recovery reconciled an incomplete operation.

An implementation response may use a result label such as `recovered_committed` to describe the recovered outcome, but the persisted event phase for new events is `recovered`.

Existing historical events and evidence are never rewritten or renamed. Readers must remain backward-compatible with already persisted event shapes and result labels.

Exactly one successful terminal event may exist for an operation:

```text
committed
or
recovered
```

A failed terminal record does not silently authorize retry under the same immutable approval unless the governing recovery contract explicitly permits an idempotent continuation.

### 6. Required event binding

Every new governed `promote` or `amend` event must bind:

```text
schema version
event ID
operation ID
event phase
action
note ID
note type
project
destination path
approved human actor
approval timestamp
validation run ID
plan SHA-256
approval SHA-256
approved or installed content SHA-256
concise basis
```

Amendment events additionally bind:

```text
prior content SHA-256
new content SHA-256
change summary
reviewed diff SHA-256
```

The exact serialized event shape remains versioned. Existing schema-version `1.0` lines remain valid.

### 7. Spec maturity vocabulary

Kaizen planning specifications use these maturity labels:

```text
draft
accepted planning contract
implemented baseline
superseded
```

Definitions:

- `draft` - not yet accepted as binding planning guidance;
- `accepted planning contract` - accepted and binding for planning or implementation, but not yet proven in a completed implementation slice;
- `implemented baseline` - accepted and verified through implementation, tests, and steward evidence;
- `superseded` - retained for history but replaced by a later accepted contract.

A spec may include a narrower qualifier when necessary, but it must map clearly to one of these four states.

The first-slice registries, validation, hammer, staging, promotion, event, and recovery specifications should become `implemented baseline` only after their Milestone 5 edits are reviewed and accepted.

Future Postgres, provider, and intelligence specifications remain `draft` or `deferred`.

### 8. Human actor identity limitation

Kaizen v0.2 accepts a trusted local actor string such as:

```text
owner.local
```

for single-owner local operation.

This is not authentication and must not be represented as proof of real-world identity.

Rules:

- the actor value is supplied only through the human-operated approval or execution boundary;
- agents may not select or fabricate the owner actor;
- the value is preserved in immutable approval and event evidence;
- shared, remote, or multi-user operation requires a later accepted identity and authentication design.

Durable identity and authentication remain a future security milestone.

### 9. Backup and remote posture

Kaizen v0.2 accepts an explicitly owner-approved local-only period for the platform and canonical vault repositories.

During that period:

- each repository remains a separate Git repository;
- clean working state is required at governed checkpoints;
- exact local commits are required;
- no remote is created or pushed without separate owner approval;
- the lack of off-machine backup is recorded as an accepted temporary risk;
- visibility and backup are reconsidered at the next implementation-roadmap acceptance gate.

`kaizen-docs` may continue using its already authorized remote.

This decision does not select public or private visibility and does not authorize creation of a platform or vault remote.

### 10. Connector and local operator posture

Typed connector tools remain preferred for inspection, validation, and narrowly classified operations when available.

Connector mutation is opportunistic, not guaranteed, because upstream safety layers may block a call before Kaizen receives it.

The guaranteed execution path is the fixed-root local human operator using the same platform enforcement code.

After one upstream mutation block:

1. verify that no approval, event, or canonical mutation occurred;
2. do not repeatedly retry equivalent mutation routes;
3. switch to the local operator or provide the owner the exact minimal command;
4. resume evidence verification after execution.

### 11. Governance proportionality

Strict gates remain mandatory for:

- canonical mutation;
- authority transitions;
- immutable plans and approvals;
- exact hashes;
- event sequencing;
- filesystem confinement;
- Git scope;
- implementation-return updates.

Equivalent ceremony is not required for:

- read-only evidence gathering;
- local retrospective analysis;
- low-risk documentation cleanup that changes no accepted contract;
- concise status mirrors of one authoritative tracker.

## Required specification updates

If accepted, update these specifications:

### `05-specs/promotion-event-schema.md`

- mark `promote` and `amend` as implemented actions;
- mark `supersede`, `correct`, and `rollback` as reserved;
- add plan and approval hash binding;
- add reviewed-diff binding for amendments;
- adopt the event-phase terminology above;
- preserve schema-version and historical-event compatibility.

### `05-specs/staging-and-promotion-workflow.md`

- rename the scope to staging and governed canonical mutation;
- distinguish first-time promotion and amendment;
- define separate plan-generation and execution approvals;
- add immutable operation evidence;
- add bounded amendment continuity rules;
- update completion-report and current-state return workflow;
- mark the implemented first-slice acceptance criteria complete after review.

### `05-specs/staging-write-wrapper-and-promotion-recovery.md`

- preserve create-only staging behavior;
- add the implemented same-path amendment replacement primitive;
- reconcile recovery phases and outcomes;
- remove statements that implementation language and canonical tooling are wholly unselected where the first slice has already selected and proven them;
- preserve future Hermes authorization as separate.

### `05-specs/kaizen-validation-gate-spec.md`

- distinguish intrinsic note validation from operation-context validation;
- preserve current validation modes and implemented behavior;
- avoid claiming standalone canonical validation proves mutation readiness.

### Registry and hammer specifications

- update maturity labels;
- close stale open questions already resolved by Decision 0008 or implementation evidence;
- preserve truly unresolved questions.

## Compatibility and migration

1. Do not rewrite historical JSONL event lines.
2. Do not change existing note or operation IDs.
3. Do not mass-rewrite canonical notes.
4. Keep first-time promotion behavior compatible.
5. Preserve current vault and staging roots.
6. Treat specification edits as planning-contract reconciliation, not runtime migration.
7. Require regression tests for any machine-readable schema change.

## Consequences

### Enables

- authoritative v0.2 acceptance after supporting specs are reconciled and audited;
- amendment-native MCP and operator tools under one contract;
- clearer reader and recovery behavior;
- honest treatment of actor identity and backup limitations;
- consistent spec maturity reporting.

### Prevents

- treating reserved actions as implemented;
- presenting `owner.local` as authentication;
- claiming connector execution is guaranteed;
- silently rewriting historical events;
- widening amendment into general overwrite;
- treating stale draft labels as harmless documentation noise.

### Defers

- multi-user identity and authentication;
- remote execution trust;
- supersedence, correction, rollback execution, delete, move, and multi-note transactions;
- platform and vault remote selection;
- Postgres, Qdrant, Hermes live integration, providers, and broad UI work.

## Acceptance gate

This decision remains proposed until the owner explicitly accepts it.

Acceptance authorizes documentation-contract reconciliation only. It does not authorize runtime implementation or the next implementation milestone.
