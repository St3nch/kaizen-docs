## Context

After accepting the v0.2 foundation, several practical conventions remained unresolved. Decision 0008 establishes the initial operating defaults for project stages, folder placement, portable paths, plugin policy, source provenance, `.obsidian` version control, private/raw-data validation, implementation feedback, and future Qdrant treatment of current-state notes.

A read-only end-to-end simulation tested these conventions against a realistic Kaizen bootstrap project. The simulation showed that the model is implementation-ready after simplifying the project-stage enum and clarifying the return loop, human provenance, source-summary relationships, and earned-folder creation.

## Decision

### 1. Project pipeline stages

Use this project-level stage enum:

```text
capture
research
spec
audit
handoff
build
```

Definitions:

- `capture` - an idea exists but has not entered structured research
- `research` - sources are being gathered, summarized, and synthesized into claims, conflicts, and implications
- `spec` - accepted direction is being translated into an implementation definition
- `audit` - the specification or handoff is under structured review
- `handoff` - an approved task packet is being prepared or is ready for a coding agent
- `build` - implementation is active in the source repository; completion evidence flows back through the governed return loop

Rules:

- `pipeline_stage` belongs on command-center and current-state notes.
- Other note types do not need the field unless a concrete workflow proves it useful.
- `pipeline_stage` describes the current dominant project phase, not a note lifecycle state, live job state, or monotonic state machine.
- A project may move backward when audit, evidence, or implementation exposes gaps.
- When command-center and current-state temporarily disagree, `current-state.pipeline_stage` is authoritative.
- `synthesis` is performed within research through source summaries, claims, conflicts, and decisions; it is not a separate v0.2 project stage.
- `operate` is deferred until a real operational project proves that a separate project stage is useful.

### 2. Canonical project folder placement

Use this minimal project root:

```text
projects/
  {project-slug}/
    command-center.md
    overview.md
    current-state.md
```

Earned folders:

```text
source-summaries/
claims/
decisions/
specs/
audits/
handoffs/
```

Placement:

| Note type | Canonical location |
|---|---|
| `command-center` | project root |
| `overview` | project root |
| `current-state` | project root |
| `source-summary` | `source-summaries/` |
| `claim` | `claims/` |
| `decision` | `decisions/` |
| `spec` | `specs/` |
| `audit` | `audits/` |
| `task-packet` | `handoffs/` |

No earned folder exists before its first real note.

A governed promotion may create a missing earned destination directory only when:

- the note type maps to that directory in the accepted placement table;
- the resolved destination remains strictly inside the canonical project root;
- traversal, device-path, reparse-point, junction, mount-point, and symlink escape checks pass;
- destination creation and file installation occur inside the same recoverable promotion workflow.

Humans may edit canonical Markdown directly when authorized. Agent-created or staged candidate artifacts require deterministic validation and human-controlled promotion. Authority transitions and governed staged promotions remain auditable.

### 3. Portable vault-root convention

Canonical documents must use vault-relative or repository-relative paths.

Allowed canonical examples:

```text
projects/searchclarity/specs/observatory-api.md
source repo: veda-ops-dev/searchclarity
repo path: docs/architecture.md
```

Local absolute paths such as `C:\dev\...` are environment hints only. They may appear in local setup docs, tool configuration, or session output, but must not serve as portable identity or cross-note reference.

Recommended symbolic root names in documentation:

```text
<KAIZEN_VAULT_ROOT>
<KAIZEN_STAGING_ROOT>
<SOURCE_REPO_ROOT>
```

### 4. Plugin policy

Kaizen v0.2 requires no Obsidian community plugin for canonical operation.

Classification:

| Plugin/capability | v0.2 policy |
|---|---|
| Dataview | optional later human read model after enough notes exist |
| Templater | defer until repeated manual note creation proves need |
| Commander | defer until stable human workflows exist |
| Tasks | defer; plain Markdown checkboxes may be used locally |
| Canvas | defer; human visual aid only |
| Kanban | defer; never canonical state |
| Excalidraw | defer; never canonical state |
| Obsidian Local REST API | avoid by default due to broad blast radius |
| Obsidian-specific MCP | evaluate only after raw filesystem workflow is proven |
| AI assistant plugins | do not install as core Kaizen infrastructure |

Every plugin dependency must be documented as human convenience, not source-of-truth logic.

### 5. `.obsidian` version-control policy

Default posture: deny by default, allow intentionally.

Track only configuration that is deliberately required for consistent human behavior and has been reviewed for portability.

Always ignore or exclude high-churn personal UI state, including:

```text
.obsidian/workspace.json
.obsidian/workspaces.json
.obsidian/cache
```

Community plugin code and settings should not be committed automatically. Each adopted plugin receives an explicit decision identifying which config files, if any, are tracked.

The canonical pipeline must remain operable when `.obsidian/` is absent.

### 6. Source URL and governed-source lifecycle

- `source_urls` identify external sources on source summaries.
- URLs remain in canonical source summaries after stable governed relationships exist.
- Accepted claims reference stable governed source artifacts through `source_docs`, normally source-summary Kaizen IDs.
- The corresponding source-summary preserves the reviewed URL and related provenance.
- A later source registry may supplement source-summary relationships but does not make source-summary IDs temporary or invalid.

If a URL changes, preserve the originally reviewed URL and add retrieval or canonical URL details rather than silently replacing history.

### 7. Raw-data and private-data validation posture

Start with hard boundaries and warning-based heuristics.

Hard errors:

- credentials, tokens, private keys, or connection strings
- known customer/private dataset markers
- binary/base64 payloads embedded in Markdown
- raw crawl or API dumps intentionally placed in canonical folders

Warnings requiring human review:

- files larger than 100 KB
- Markdown tables longer than 100 data rows
- JSON/code blocks longer than 200 lines
- repeated email/phone-like patterns
- pasted transcripts or logs longer than 1,000 lines

These thresholds are initial guardrails, not permanent doctrine. Tune them using real false-positive and false-negative evidence.

No scanner override may permit secrets into canonical content.

### 8. Completion reports and implementation return loop

A task packet's `## Completion Report` is populated after implementation through a governed amendment when the task packet is under staged/promotion control.

The completion report records evidence such as:

- deliverables produced;
- tests executed;
- deviations;
- failures;
- discoveries;
- follow-up work.

The completion report itself carries no authority.

Findings that change project intelligence must become new governed notes or governed amendments to existing notes, most commonly claims, decisions, specs, audits, follow-up task packets, and `current-state.md`.

Those artifacts follow the normal validation, review, and promotion workflow. After implementation, the project may return from `build` to `spec`, `research`, or another earlier stage for the next loop.

### 9. Future Qdrant indexing of current-state notes

Exclude `current-state` notes from default semantic retrieval at first.

Reason:

- they are time-sensitive snapshots;
- stale snapshots can contaminate context packs;
- their value is usually direct navigation, not durable semantic knowledge.

When Qdrant exists, current-state notes may be placed in a separate collection or retrieved only through explicit filters such as:

```text
type = current-state
project = <project>
```

They must not appear in default cross-project semantic search until retrieval evaluation proves the benefit.

## Consequences

### Benefits

- closes the remaining first-slice operating ambiguity;
- keeps canonical operation plugin-independent;
- gives each initial note type one obvious destination;
- preserves portable paths and source provenance;
- establishes a governed implementation-feedback loop without creating a premature note type;
- prevents volatile current-state content from polluting semantic retrieval;
- establishes practical validation guardrails without pretending thresholds are fully researched.

### Costs

- plugin conveniences arrive later;
- validators must support warnings and errors separately;
- project stages may move backward and therefore cannot be treated as a monotonic workflow engine;
- the promotion implementation must safely create earned folders under strict final-path confinement.

## Rejected alternatives

### Keep separate `synthesis` and `operate` stages in v0.2

Rejected because the dry run showed that synthesis belongs within research and no first-slice operational state requires `operate`.

### Store all note types directly in project root

Rejected because folder clarity becomes poor as projects grow.

### Require plugins during initial vault creation

Rejected because canonical operation must remain raw-file portable.

### Remove URLs after governed source IDs exist

Rejected because URLs remain useful human provenance and recovery information.

### Treat completion reports as authority

Rejected because implementation reports are evidence. Binding changes require governed follow-up notes.

### Index current-state notes by default

Rejected initially because freshness and staleness are more important than semantic similarity for these notes.

### Commit all `.obsidian` settings

Rejected because personal workspace state and plugin configuration create hidden infrastructure and churn.

## Deferred questions

- Whether `operate` should be added after a real operational project walkthrough.
- Whether `capture` and `research` should merge for very small projects.
- Whether raw-data warning thresholds should be lower for Qdrant-indexed content.
- Exact `.gitignore` entries when the production vault is created.

## Acceptance basis

Decision 0008 is accepted because:

1. the v0.2 foundation is accepted;
2. document-contract research found no conflicting operating model;
3. a realistic project chain was simulated through the stages, folders, completion report, and current-state return loop;
4. the pipeline-stage enum was revised from simulation evidence;
5. the remaining deterministic and human-review boundaries are explicit.
