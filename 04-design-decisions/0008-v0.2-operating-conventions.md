# Decision 0008 - Kaizen v0.2 Operating Conventions

Status: proposed
Date: 2026-06-04
Related files:

## Context

After accepting the v0.2 foundation, several practical conventions remained unresolved. Most are design choices rather than open research questions. This proposal gives Kaizen a coherent default for pipeline stages, folder placement, portable paths, plugin policy, source URL retention, `.obsidian` version control, private/raw-data validation, and future Qdrant treatment of current-state notes.

This decision remains proposed until human review and, where useful, Claude's independent recommendation.

## Proposed decision

### 1. Pipeline stages

Use this project-level stage enum:

```text
capture
research
synthesis
spec
audit
handoff
build
operate
```

Definitions:

- `capture` - idea exists but has not entered structured research
- `research` - sources are being gathered and summarized
- `synthesis` - claims, conflicts, and implications are being formed
- `spec` - accepted direction is being translated into an implementation definition
- `audit` - the spec or handoff is under structured review
- `handoff` - an approved task packet is being prepared or is ready for a coding agent
- `build` - implementation is active in the source repository
- `operate` - the project is live or maintained; new changes re-enter earlier stages as needed

Rules:

- `pipeline_stage` belongs on command-center and current-state notes.
- Other note types do not need the field unless a concrete workflow proves it useful.
- A project may move backward when audit, evidence, or implementation exposes gaps.
- `pipeline_stage` is a human-readable project snapshot, not live job state.

### 2. Canonical project folder placement

Use this minimal project root:

```text
projects/
  {project-slug}/
    command-center.md
    overview.md
    current-state.md
```

Earned folders:

```text
source-summaries/
claims/
decisions/
specs/
audits/
handoffs/
```

Placement:

| Note type | Canonical location |
|---|---|
| `command-center` | project root |
| `overview` | project root |
| `current-state` | project root |
| `source-summary` | `source-summaries/` |
| `claim` | `claims/` |
| `decision` | `decisions/` |
| `spec` | `specs/` |
| `audit` | `audits/` |
| `task-packet` | `handoffs/` |

No earned folder exists before its first real note.

### 3. Portable vault-root convention

Canonical documents must use vault-relative or repository-relative paths.

Allowed canonical examples:

```text
projects/searchclarity/specs/observatory-api.md
source repo: veda-ops-dev/searchclarity
repo path: docs/architecture.md
```

Local absolute paths such as `C:\dev\...` are environment hints only. They may appear in local setup docs, tool configuration, or session output, but must not serve as portable identity or cross-note reference.

Recommended symbolic root names in documentation:

```text
<KAIZEN_VAULT_ROOT>
<KAIZEN_STAGING_ROOT>
<SOURCE_REPO_ROOT>
```

### 4. Plugin policy

Kaizen v0.2 requires no Obsidian community plugin for canonical operation.

Classification:

| Plugin/capability | v0.2 policy |
|---|---|
| Dataview | optional later human read model after enough notes exist |
| Templater | defer until repeated manual note creation proves need |
| Commander | defer until stable human workflows exist |
| Tasks | defer; plain Markdown checkboxes may be used locally |
| Canvas | defer; human visual aid only |
| Kanban | defer; never canonical state |
| Excalidraw | defer; never canonical state |
| Obsidian Local REST API | avoid by default due to broad blast radius |
| Obsidian-specific MCP | evaluate only after raw filesystem workflow is proven |
| AI assistant plugins | do not install as core Kaizen infrastructure |

Every plugin dependency must be documented as human convenience, not source-of-truth logic.

### 5. `.obsidian` version-control policy

Default posture: deny by default, allow intentionally.

Track only configuration that is deliberately required for consistent human behavior and has been reviewed for portability.

Always ignore or exclude high-churn personal UI state, including:

```text
.obsidian/workspace.json
.obsidian/workspaces.json
.obsidian/cache
```

Community plugin code and settings should not be committed automatically. Each adopted plugin receives an explicit decision identifying which config files, if any, are tracked.

The canonical pipeline must remain operable when `.obsidian/` is absent.

### 6. Source URL lifecycle

Before a stable source registry exists:

- `source_urls` may identify external sources on source summaries.
- URLs remain in the canonical note after stable source IDs are assigned.
- `source_docs` becomes the machine relationship field once a stable source note/registry object exists.
- URLs remain human-readable provenance and recovery information.

Accepted claims should reference stable `source_docs`; URLs alone are insufficient for accepted authority.

If a URL changes, preserve the originally reviewed URL and add retrieval/canonical URL details rather than silently replacing history.

### 7. Raw-data and private-data validation posture

Start with hard boundaries and warning-based heuristics.

Hard errors:

- credentials, tokens, private keys, or connection strings
- known customer/private dataset markers
- binary/base64 payloads embedded in Markdown
- raw crawl or API dumps intentionally placed in canonical folders

Warnings requiring human review:

- files larger than 100 KB
- Markdown tables longer than 100 data rows
- JSON/code blocks longer than 200 lines
- repeated email/phone-like patterns
- pasted transcripts or logs longer than 1,000 lines

These thresholds are initial guardrails, not permanent doctrine. Tune them using real false-positive and false-negative evidence.

No scanner override may permit secrets into canonical content.

### 8. Future Qdrant indexing of current-state notes

Exclude `current-state` notes from default semantic retrieval at first.

Reason:

- they are time-sensitive snapshots
- stale snapshots can contaminate context packs
- their value is usually direct navigation, not durable semantic knowledge

When Qdrant exists, current-state notes may be placed in a separate collection or retrieved only through explicit filters such as:

```text
type = current-state
project = <project>
```

They must not appear in default cross-project semantic search until retrieval evaluation proves the benefit.

## Consequences

### Benefits

- closes most remaining v0.2 design ambiguity
- keeps canonical operation plugin-independent
- gives each initial note type one obvious destination
- preserves portable paths
- retains useful source provenance
- prevents volatile current-state content from polluting semantic retrieval
- establishes practical validation guardrails without pretending thresholds are fully researched

### Costs

- plugin conveniences arrive later
- source URLs and stable IDs create some provenance duplication
- validators must support warnings and errors separately
- projects in active operation may need stage backtracking rules

## Rejected alternatives

### Store all note types directly in project root

Rejected because folder clarity becomes poor as projects grow.

### Require plugins during initial vault creation

Rejected because canonical operation must remain raw-file portable.

### Remove URLs after source IDs exist

Rejected because URLs remain useful human provenance and recovery information.

### Index current-state notes by default

Rejected initially because freshness and staleness are more important than semantic similarity for these notes.

### Commit all `.obsidian` settings

Rejected because personal workspace state and plugin configuration create hidden infrastructure and churn.

## Open questions

- Whether `operate` belongs in v0.2 or should be deferred until the first live project walkthrough.
- Whether `capture` and `research` should be merged for very small projects.
- Whether raw-data warning thresholds should be lower for Qdrant-indexed content.
- Exact `.gitignore` entries once the production vault is created.

## Acceptance recommendation

Accept after:

1. human review of the stage enum and folder placement
2. Claude's independent document-contract research confirms no conflict
3. a dry-run mapping of one real project through the stages and folders
